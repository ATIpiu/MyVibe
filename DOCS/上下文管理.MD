# VibeCoding工具 代码记忆管理系统 详细设计文档 v1.0

**2026年3月6日**

> 核心架构：模块化开发 + 函数级摘要 + JSON管理 + 按需加载

---

## 1. 系统概述

### 1.1 设计目标

本系统的核心目标是为 VibeCoding 工具提供一套完整的代码记忆管理机制，使得 LLM 在生成、修改、重构代码时，能够精准地获取所需的上下文信息，而不必将全量代码放入上下文窗口。

系统采用三层架构：索引层（轻量函数摘要）、详情层（完整源码）、智能调度层（LLM 主动请求所需信息）。

### 1.2 设计原则

- **最小记忆原则**：默认只存储函数签名、一句话用途、输入输出类型，不存完整代码
- **按需加载原则**：LLM 在执行任务前主动思考需要哪些具体信息，系统再按需拉取
- **自动同步原则**：写入/修改/删除操作自动触发记忆更新，无需手动维护
- **唯一标识原则**：每个函数用 `相对路径:函数名` 作为全局唯一 key
- **版本可追溯原则**：每次修改记录版本号和时间戳，支持回滚分析

### 1.3 整体架构图

系统分为三个核心层次，各层职责明确：

```
+------------------------------------------------------------------+
|                    智能调度层 (Orchestration Layer)               |
|  LLM 接收任务 → 思考需要什么信息 → 发起查询请求                    |
+------------------------------------------------------------------+
         |              |              |              |
    get_function   get_module   get_callers   search_by_purpose
         |              |              |              |
+------------------------------------------------------------------+
|                     索引层 (Index Layer)                          |
|  functions.json │ modules.json │ call_graph.json                  |
|  轻量级摘要 │ 全局可搜索 │ 关系图谱                                |
+------------------------------------------------------------------+
         |              |              |
    read_source   read_docstring   read_tests
         |              |              |
+------------------------------------------------------------------+
|                    详情层 (Detail Layer)                          |
|  实际源码文件 │ 完整 docstring │ 单元测试                          |
+------------------------------------------------------------------+
```

## 2. 记忆存储结构设计

### 2.1 文件组织

记忆系统的存储文件结构如下：

```
.vibecoding/
  memory/
    functions.json        # 函数级记忆索引（核心）
    modules.json          # 模块级元信息
    call_graph.json       # 函数调用关系图
    metadata.json         # 全局元数据
    history/              # 版本历史快照（可选）
      functions_20260305.json
      functions_20260306.json
```

### 2.2 functions.json —— 函数记忆索引（核心）

这是整个系统最重要的文件。每个函数的 key 采用 `相对路径:函数名` 格式，确保全局唯一。字段设计如下：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| module | string | 文件相对路径，如 `core/parser.py` |
| name | string | 函数名 |
| qualname | string | 完整限定名，如 `ClassName.method_name` |
| signature | string | 完整函数签名，包含类型注解 |
| input_schema | object | 结构化的输入参数描述（类型 + 一句话描述） |
| return_schema | object | 结构化的返回值描述（类型 + 一句话描述） |
| purpose | string | 一句话用途说明，用于全局搜索和 LLM 快速理解 |
| docstring_summary | string | 原始 docstring 的摘要（比 purpose 更详细） |
| calls | string[] | 本函数调用的其他函数的唯一 key 列表 |
| called_by | string[] | 调用本函数的其他函数的唯一 key 列表 |
| last_modified | ISO 8601 | 最后修改时间戳 |
| version | integer | 版本号，每次修改自动递增 |
| access_count | integer | 被 LLM 查询的次数，用于重要度计算和清理 |
| last_access | ISO 8601 | 最后一次被查询的时间 |
| tags | string[] | 自定义标签，如 `["parsing", "config", "io"]`，支持语义搜索 |

> **设计决策：为什么用结构化 input_schema 而不是自然语言？**
>
> 自然语言描述如"接受一个路径和可选配置"容易被 LLM 误解类型。结构化描述 `{"path": {"type": "str \| Path"}}` 能大幅降低幻觉和类型错误。

### 2.3 完整 JSON 示例

```json
{
  "core/parser.py:parse_user_config": {
    "module": "core/parser.py",
    "name": "parse_user_config",
    "qualname": "parse_user_config",
    "signature": "def parse_user_config(path: str | Path, default: dict | None = None) -> dict:",
    "input_schema": {
      "path": {
        "type": "str | Path",
        "description": "配置文件路径"
      },
      "default": {
        "type": "dict | None",
        "description": "缺省配置，用于合并"
      }
    },
    "return_schema": {
      "type": "dict",
      "description": "解析后的配置字典，已合并默认值"
    },
    "purpose": "读取并解析用户配置文件，支持 TOML/YAML/JSON",
    "docstring_summary": "安全解析用户配置文件，返回合并后的配置字典",
    "calls": [
      "utils/path.py:ensure_path_exists",
      "config/defaults.py:get_default_config"
    ],
    "called_by": [
      "main.py:init_app"
    ],
    "last_modified": "2026-03-05T14:30:22Z",
    "version": 4,
    "access_count": 23,
    "last_access": "2026-03-06T03:12:45Z",
    "tags": ["parsing", "config", "io"]
  }
}
```

### 2.4 modules.json —— 模块级元信息

每个 Python 文件对应一条模块记录，用于 LLM 快速了解模块职责而无需查看每个函数：

```json
{
  "core/parser.py": {
    "purpose": "负责所有配置文件的读写与格式转换",
    "main_functions": ["parse_user_config", "save_config", "validate_config"],
    "exports": ["parse_user_config", "save_config"],
    "dependencies_external": ["toml", "pyyaml"],
    "dependencies_internal": ["utils/path.py", "config/defaults.py"],
    "last_modified": "2026-03-05T14:30:00Z",
    "function_count": 5
  }
}
```

### 2.5 call_graph.json —— 调用关系图

独立存储函数间的调用关系，便于全局重构和影响分析：

```json
{
  "edges": [
    {
      "caller": "main.py:init_app",
      "callee": "core/parser.py:parse_user_config",
      "call_type": "direct"
    },
    {
      "caller": "core/parser.py:parse_user_config",
      "callee": "utils/path.py:ensure_path_exists",
      "call_type": "direct"
    }
  ],
  "last_updated": "2026-03-06T05:20:11Z"
}
```

> **为什么单独存储调用图？**
>
> 如果只在 functions.json 的 calls/called_by 字段中记录，修改一个函数时需要同时更新多个其他函数的记录。独立的 call_graph.json 可以原子化更新，然后定期同步回 functions.json。

## 3. 智能调度层：按需加载机制

这是本系统的核心创新点。LLM 在接受一个任务后，不是直接开始编码，而是先进入"思考阶段"，分析需要哪些具体信息，然后通过工具调用获取。

### 3.1 完整工作流程

以下是一个典型的任务执行流程：

1. **任务接收**：LLM 收到用户指令，如"给 parse_user_config 加上环境变量覆盖功能"
2. **思考阶段**：LLM 分析这个任务需要了解什么——当前函数的完整代码、它调用的函数、谁在调用它
3. **信息请求**：LLM 发起一系列 tool calls 获取所需信息
4. **信息注入**：系统返回具体信息，加入上下文窗口
5. **代码生成**：LLM 基于充分的上下文生成代码
6. **记忆同步**：写入工具自动触发记忆更新

### 3.2 思考阶段的 Prompt 设计

这是触发按需加载的关键。在 System Prompt 中嵌入以下指令：

```markdown
## 代码任务执行协议

当你收到一个代码相关任务时，必须先执行以下思考步骤：

1. 识别任务类型：新增功能 / 修改现有代码 / 重构 / Bug修复
2. 列出需要了解的信息：
   - 哪些现有函数需要查看完整代码？
   - 哪些模块的职责需要了解？
   - 是否需要查看调用关系？
   - 是否需要查看单元测试？
3. 调用对应的查询工具获取信息
4. 确认信息充分后再开始编码
```

### 3.3 查询工具集设计

LLM 可以调用以下工具来获取不同粒度的信息：

| 工具名称 | 输入 | 输出 / 用途 |
|----------|------|-------------|
| get_function_summary | function_key | 返回函数的索引信息（签名、用途、输入输出） |
| get_function_source | function_key | 返回函数的完整源码（从详情层拉取） |
| get_module_info | module_path | 返回模块的职责、主要函数列表、依赖 |
| get_callers | function_key | 返回所有调用该函数的函数列表（影响分析） |
| get_callees | function_key | 返回该函数调用的所有函数列表（依赖分析） |
| search_functions | query / tags | 按关键词或标签搜索函数，返回匹配的摘要列表 |
| get_module_source | module_path | 返回整个模块的完整源码（重量级操作） |
| get_function_tests | function_key | 返回与该函数相关的单元测试代码 |

### 3.4 工具调用场景示例

#### 场景 A：修改现有函数

用户指令："给 parse_user_config 加上环境变量覆盖功能"

**LLM 思考过程：**

```
→ 这是修改现有函数，我需要：
  1. 查看当前函数的完整代码
  2. 查看它调用了哪些函数（可能需要修改调用方式）
  3. 查看谁在调用它（返回值类型改变会影响调用方）
```

**工具调用序列：**

```python
get_function_source("core/parser.py:parse_user_config")
get_callees("core/parser.py:parse_user_config")
get_callers("core/parser.py:parse_user_config")

# 信息充分后，开始生成修改后的代码
```

#### 场景 B：新增功能

用户指令："新增一个配置文件格式转换功能，支持 TOML 转 JSON"

**LLM 思考过程：**

```
→ 这是新增功能，我需要：
  1. 查看现有的配置相关模块，避免重复实现
  2. 确定这个新函数应该放在哪个模块
  3. 了解现有的输入输出范式，保持一致性
```

**工具调用序列：**

```python
search_functions(query="config", tags=["parsing"])
get_module_info("core/parser.py")
get_function_source("core/parser.py:parse_user_config")
# → 参考现有函数的风格和范式
```

#### 场景 C：重构

用户指令："把所有 config 相关的函数抽取到一个新模块 config_manager.py"

**LLM 思考过程：**

```
→ 这是重构任务，影响范围很大，我需要：
  1. 搜索所有 config 相关函数
  2. 查看每个函数的完整代码
  3. 查看每个函数的所有调用方（需要更新 import）
  4. 查看相关测试文件
```

**工具调用序列：**

```python
search_functions(tags=["config"])
# → 找到 5 个相关函数

get_function_source(每个函数) × 5
get_callers(每个函数) × 5
get_function_tests(每个函数) × 5

# 影响分析完成后，生成重构方案和代码
```

## 4. 自动同步机制

记忆系统的生命力在于自动同步。所有代码操作（写入、修改、删除）都必须自动触发记忆更新。

### 4.1 写入操作的同步流程

```
用户指令 "创建 xxx 函数"
    │
    ├─ LLM 思考阶段（查询工具）
    │
    ├─ LLM 生成代码
    │
    ├─ 调用 write_file 工具写入文件
    │     │
    │     ├─ 写入源码到磁盘
    │     │
    │     ├─ 自动解析函数签名（AST）
    │     │     ├─ 提取 name, signature, input_schema,
    │     │     │   return_schema, docstring
    │     │     ├─ 检测函数调用关系（静态分析）
    │     │     └─ 生成唯一 key
    │     │
    │     ├─ 更新 functions.json
    │     ├─ 更新 modules.json
    │     └─ 更新 call_graph.json
    │
    └─ 返回结果给 LLM
```

### 4.2 修改操作的同步流程

1. **读取旧版本**：从 functions.json 获取当前版本号和信息
2. **写入新代码**：更新源码文件
3. **重新解析**：用 AST 重新提取函数信息
4. **Diff 检测**：对比新旧信息，只更新变化的字段
5. **版本递增**：`version += 1`，更新 `last_modified`
6. **级联更新**：如果函数签名改变，更新所有 `called_by` 中引用它的函数

### 4.3 删除操作的同步流程

删除是最容易产生"尸体记忆"的操作，必须彻底清理：

1. **删除源码**：从文件中移除函数代码
2. **删除索引**：从 functions.json 移除对应条目
3. **清理调用图**：从 call_graph.json 移除所有相关边
4. **更新被影响的记录**：更新其他函数的 `calls / called_by` 字段
5. **更新模块信息**：更新 modules.json 中的 `main_functions` 列表
6. **孤立检测**：检查是否有其他函数仍在调用已删除函数，发出警告

> **关键实现细节：孤立引用检测**
>
> 删除函数 A 后，如果函数 B 的 calls 中仍包含 A，系统应输出警告：`"core/parser.py:func_b 仍在调用已删除的 core/utils.py:func_a"`。这可以有效防止删除不彻底导致的运行时错误。

## 5. AST 自动解析引擎

记忆的自动生成依赖于 Python AST（抽象语法树）解析。以下是解析引擎的核心功能：

### 5.1 解析器职责

| 解析内容 | 实现方式 |
|----------|----------|
| 函数签名 | `ast.FunctionDef / ast.AsyncFunctionDef` → 提取 name, args, returns, defaults |
| 类型注解 | `ast.arg.annotation` → 转为字符串类型描述 |
| Docstring | `ast.get_docstring()` → 提取第一行作为 summary |
| 函数调用 | 遍历 `ast.Call` 节点，解析 `func.attr` 和 `func.id` |
| Import 关系 | `ast.Import / ast.ImportFrom` → 构建函数名到模块的映射 |
| 类方法 | `ast.ClassDef` 内的 FunctionDef → `qualname = ClassName.method_name` |

### 5.2 解析器伪代码

```python
def parse_module(file_path: str) -> ModuleInfo:
    source = read_file(file_path)
    tree = ast.parse(source)

    functions = {}
    imports = extract_imports(tree)  # 构建名称→模块映射

    for node in ast.walk(tree):
        if isinstance(node, (ast.FunctionDef, ast.AsyncFunctionDef)):
            func_info = FunctionInfo(
                name=node.name,
                qualname=get_qualname(node),
                signature=extract_signature(node),
                input_schema=extract_input_schema(node),
                return_schema=extract_return_schema(node),
                docstring_summary=get_docstring_first_line(node),
                calls=extract_calls(node, imports),
            )
            key = f"{relative_path}:{func_info.qualname}"
            functions[key] = func_info

    return ModuleInfo(functions=functions, ...)
```

### 5.3 调用关系解析的难点与解决方案

静态分析函数调用关系是最复杂的部分，因为 Python 是动态语言。以下是建议的分级策略：

- **第一级（必须实现）**：解析直接的 `module.function()` 调用和同模块内的函数调用
- **第二级（建议实现）**：解析 `from xxx import yyy` 后的直接调用 `yyy()`
- **第三级（可选）**：解析通过变量赋值后的间接调用，如 `fn = get_parser(); fn()`

> **实用建议**：不要追求 100% 的调用关系准确率。第一级和第二级已经能覆盖绝大多数场景。剩余的可以通过 LLM 在思考阶段主动查询来补充。

## 6. 记忆清理与维护策略

### 6.1 尸体记忆检测

尸体记忆是指 functions.json 中存在但实际源码已不存在的记录。定期执行以下检查：

1. **文件存在性检查**：遍历 functions.json，检查每个条目的模块文件是否存在
2. **函数存在性检查**：用 AST 解析模块文件，检查函数是否仍存在
3. **签名一致性检查**：对比记忆中的签名和实际代码的签名是否匹配
4. **死链接检查**：检查 `calls/called_by` 中引用的函数是否都存在

### 6.2 低频函数清理

随着项目增长，functions.json 会越来越大。建议实现基于使用频率的清理策略：

```python
# 清理策略伪代码
for func_key, func_info in functions.items():
    days_since_access = now - func_info.last_access
    if (func_info.access_count == 0
        and days_since_access > 30
        and len(func_info.called_by) == 0):
        # 标记为"可清理"，不直接删除
        mark_for_cleanup(func_key)
```

> **重要原则**：清理只是标记，不自动删除。最终删除需要用户确认。

### 6.3 完整性自检报告

建议实现一个 `memory_health_check` 命令，输出以下报告：

```
=== 记忆系统健康检查报告 ===

总函数数：87
总模块数：12
调用图边数：156

✅ 文件存在性：87/87 通过
✅ 函数存在性：87/87 通过
⚠️  签名不一致：2 个
   - core/parser.py:parse_user_config (参数改变)
   - utils/path.py:normalize (返回类型改变)
⚠️  死链接：1 个
   - core/parser.py:parse_user_config → 已删除的 config/legacy.py:old_parse
ℹ️  30天未访问且无调用方：5 个（可考虑清理）
```

## 7. 上下文窗口优化策略

记忆系统的核心价值在于减少上下文窗口的浪费。以下是不同任务场景下的上下文加载策略：

### 7.1 上下文分级加载

| 任务类型 | 必须加载 | 建议加载 | 估算 Token |
|----------|----------|----------|------------|
| 新增函数 | 模块信息 + 同模块函数摘要 | 相似函数源码（1-2个） | 500-2000 |
| 修改函数 | 目标函数源码 + 调用关系 | 调用方源码 + 测试 | 1000-4000 |
| Bug修复 | 目标函数源码 + 测试 + 调用链 | 相关模块源码 | 2000-6000 |
| 重构 | 所有相关函数源码 + 全部调用方 | 测试 + 模块全源码 | 5000-15000 |

### 7.2 上下文预算控制

建议为每次任务设置上下文预算上限。当查询工具返回的信息超过预算时，采取以下策略：

- **截断策略**：优先加载目标函数源码，其次是直接调用方/被调用方，最后是间接关联
- **摘要降级**：当预算不足时，对边缘函数只加载摘要而非源码
- **分批加载**：对于重构等大任务，可以分多轮对话完成，每轮处理一部分函数

## 8. 分阶段实现路线图

### Phase 1：核心基础（建议 1-2 周）

**目标**：让记忆系统跑起来，能自动维护函数索引。

- functions.json 数据结构定义
- 基于 AST 的函数签名自动提取
- 写入/修改/删除时自动同步 functions.json
- 唯一 key 策略：`相对路径:函数名`
- 基本的 `get_function_summary` 和 `get_function_source` 查询工具

### Phase 2：智能调度（建议 1-2 周）

**目标**：让 LLM 能够主动思考并按需获取信息。

- System Prompt 中加入"代码任务执行协议"

## 9. 已知反模式与规避方案

| 反模式 | 为什么危险 | 规避方案 |
|--------|------------|----------|
| 只存自然语言描述 | LLM 对"接受一个参数"的理解很随意 | 用结构化 input_schema 明确类型 |
| 不加版本号 | 无法知道记忆是否过时 | 每次修改自动递增 version |
| 全量加载到上下文 | 超过 50 个函数后上下文窗口爆炸 | 索引+按需加载三层架构 |
| 只用函数名做 key | 不同模块的同名函数冲突 | 用 `路径:函数名` 做唯一标识 |
| 删除时只删代码 | 记忆中的死引用导致幻觉 | 删除时级联清理所有相关记录 |
| 手动维护记忆 | 人一定会忘记更新 | 写入工具自动触发 AST 解析+更新 |
| 记忆无上限增长 | JSON 文件过大时解析变慢 | 定期清理 + 低频函数标记机制 |
| 不存 docstring | 一句话简介丢失太多细节 | 同时存储 purpose + docstring_summary |

### 10.1 写入工具接口

```python
def write_file(
    file_path: str,
    content: str,
    auto_sync: bool = True

    """
```

```python
    function_key: str,
```

## 11. 总结与关键原则

本设计文档定义了一套完整的代码记忆管理系统，核心思想可以归结为以下几点：

1. **三层架构是基础**：索引层提供快速概览，详情层提供完整信息，智能调度层决定加载什么。
2. **按需加载是核心**：LLM 先思考需要什么信息，再通过工具获取，而不是把所有代码塞进上下文。
3. **自动同步是生命线**：所有代码操作自动触发记忆更新，人不需要手动维护。
4. **结构化描述减少幻觉**：用结构化的 `input_schema/return_schema` 替代自然语言，大幅降低 LLM 误解。
5. **分阶段落地最实际**：从 Phase 1 开始，先让核心跑起来，再逐步添加高级特性。

### 最终目标

让 LLM 像一个经验丰富的开发者一样工作——不是记住所有代码，而是知道代码在哪里、做什么用、谁在用它，需要时能快速查阅。
